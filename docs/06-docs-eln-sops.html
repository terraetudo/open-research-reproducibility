Chapter 6 — Documentation and protocols

Keeping decisions visible

Chapter 6

Documentation and protocols

Documentation is how research decisions remain visible after the meeting, the analysis session, the handover, or the paper submission. It should help someone understand what was done, why it was done, and how to continue or check the work.

In this chapter

This chapter focuses on research documentation: the records that help a project remain understandable, reproducible and reusable. It covers READMEs, data dictionaries, metadata, decision logs, changelogs, environment notes, ELNs, SOPs, protocols, preregistration and reporting guidelines.

It deliberately does not repeat folder structures and file naming from Chapter 3, data-management responsibilities from Chapter 4, or tidy-data rules from Chapter 5. Instead, it asks: what needs to be written down, where, and for whom?

Learning objectives

By the end of this chapter, you should be able to:

  • explain why documentation is central to reproducibility, continuity and handover;
  • distinguish between different documentation artefacts and their purposes;
  • decide when to use a README, codebook, metadata record, decision log, changelog, ELN, SOP or protocol;
  • recognise documentation as a way of reducing project risk and “bus factor”;
  • apply proportionate documentation practices in everyday research workflows.

Why documentation matters

Documentation is often treated as an administrative extra. In reproducible research, it is part of the research method. It provides evidence of how choices were made and helps others understand whether those choices are justified.

A common problem is not simply that researchers document too little. It is that documentation has no clear purpose. Some projects have almost nothing written down; others have information scattered across emails, Teams chats, notebooks, Word documents, old protocols and people’s memories. Both situations are fragile.

Under-documenting and over-documenting are both risks

Documentation is useful when it is findable, current and purposeful. It is less useful when it is either missing or spread across too many places without a clear owner.

Good documentation helps to:

  • track what happened;
  • make decisions visible;
  • support new team members;
  • reduce reliance on memory;
  • make handover easier;
  • support audit, peer review and reuse;
  • explain deviations, exclusions and constraints.

The Turing Way and Good Enough Practices both emphasise that documentation should make it possible for someone else — or your future self — to reconstruct the work, not simply remember that something was done1,2.

The bus factor

The bus factor is a useful way to think about documentation. It asks: how many people would need to become unavailable before the project could no longer continue?

In research projects, the problem is often less dramatic than a bus: a student finishes, a research fellow moves institution, a collaborator changes role, or a data manager is on leave. The effect is the same if key knowledge only exists in one person’s head.

Activity: what would you need to know?

Imagine that a key member of your project team becomes unavailable tomorrow.

What would you need to know?

Consider:

  • where the data are stored;
  • who has access;
  • which files are raw, derived or final;
  • what approvals or data-sharing agreements apply;
  • how exclusions or analysis decisions were made;
  • what software, packages and versions were used;
  • which outputs are current;
  • what still needs to be done.

Documentation should be proportionate

Not every project needs every type of document. A short student project may need a README, a data dictionary and a few notes on analysis decisions. A multi-partner project using sensitive linked data may need a DMP, README, data dictionary, metadata record, decision log, changelog, SOPs, governance documents and access-control records.

The principle is proportionality:

  • document decisions that affect interpretation, reuse or trust;
  • keep documentation close to the relevant files;
  • avoid duplicating the same information in multiple places;
  • make ownership and update frequency clear;
  • use simple formats where possible.

A documentation map

Different documentation artefacts answer different questions.

Documentation artefact Main question it answers Main audience
README What is this project or dataset, and how do I navigate it? New collaborator, reviewer, future you
Data dictionary / codebook What does each variable mean? Analyst, reviewer, secondary user
Metadata What is this dataset, where did it come from, and how should it be cited/discovered? Repository, data steward, future user
Decision log Why was this choice made? Project team, reviewer, future analyst
Changelog What changed, when, and by whom? Project team, future user
Environment notes What software context is needed to rerun the work? Analyst, future user
ELN What happened during active work? Project team
SOP How should this repeatable task be done? Team member, new staff
Protocol / preregistration What was planned before data collection or analysis? Reviewer, future reader, research team

When to use each document — quick guide

Situation Useful documentation
A new person joins the project README
A variable needs explaining Data dictionary or codebook
A dataset needs citing, archiving or discovery Metadata
A choice needs justifying later Decision log
A file, output or document changes Changelog
Code needs to be rerun in the same environment Environment notes
A task must be done consistently SOP
A study or analysis plan needs to be visible before results Protocol or preregistration

Start with a simple README

A README is the first place someone should look when they enter a project folder, repository or dataset. It does not need to be long. It should orientate the reader.

A project README should usually answer:

  1. What is this project? Purpose, scope, key outputs.
  2. What does it contain? Data files, scripts, documents and folder structure.
  3. How does it work? How to run the analysis or navigate the materials.
  4. What decisions matter? Versions, exclusions, constraints and key assumptions.
  5. What are the rules? Licence, sharing conditions, citation and contact details.

For a dataset, a README may also include data collection methods, variable descriptions, missing-value codes, software requirements and restrictions on reuse3,4.

Keep it findable

Put the README at the highest relevant level: project root, data folder or dataset deposit. A hidden README is not doing its job.

Data dictionaries and codebooks

A data dictionary or codebook describes the variables in a dataset. This is different from a README. A README orientates the project or dataset as a whole; a data dictionary explains the internal structure of the data.

A useful data dictionary includes:

  • variable name;
  • clear variable description;
  • unit of measurement;
  • expected data type;
  • allowed values or categories;
  • coding scheme;
  • missing-value codes;
  • source variable or source dataset;
  • derivation or transformation, where relevant.

This is particularly important for administrative, survey, cohort and linked datasets, where variable meanings may depend on coding rules, collection context, eligibility, geography, time period or provider systems5,6.

Metadata: dataset-level context

Metadata describe the dataset as a research object. They support discovery, citation, interpretation, archiving and reuse.

At minimum, metadata should capture:

  • title and description;
  • creators and contributors;
  • data source and collection context;
  • dates, versions and time period covered;
  • geography or population, where relevant;
  • inclusion and exclusion criteria;
  • provenance: how the data were generated, processed or cleaned;
  • access conditions and restrictions;
  • licence or reuse conditions;
  • persistent identifier, if available.

Metadata standards vary by field and repository. Examples include Dublin Core, DataCite, DDI, SDMX and ISO 19115 for geographic information7–11. The Research Data Alliance Metadata Standards Catalog can help identify relevant standards12. In practice, most researchers do not need to master these standards, but they do need to understand what metadata are trying to achieve: enough context for the data to be found, understood and reused appropriately13,14.

Metadata is not only for open data

Restricted or sensitive data still need metadata. Even when the data themselves cannot be shared openly, metadata can explain what exists, who controls access, what restrictions apply, and how legitimate access requests are handled.

Decision logs

A decision log records choices that may matter later. It is especially useful when the “right” choice is not obvious, when decisions are made under constraints, or when alternative choices could affect results.

Decision logs are useful for:

  • key analytical choices;
  • exclusions and deviations;
  • parameter changes;
  • changes to inclusion criteria;
  • changes to exposure or outcome definitions;
  • trade-offs caused by data quality, governance, ethics or feasibility;
  • choices made after discussion with collaborators or PPIE contributors.

Example:

Date Decision Rationale Impact Who
2025-02-12 Excluded records with missing postcode Exposure assignment required geocoding Reduced sample size by 3.2% JC
2025-02-18 Defined heatwave as ≥3 consecutive days above 95th percentile Aligned with agreed threshold; sensitivity tested Affected exposure prevalence estimates JC + PH
2025-02-25 Used 100 m buffer for greenspace instead of 300 m 300 m caused high spatial overlap; reduced precision Changed exposure metric; documented in methods JC

A decision log can be a simple Markdown file, spreadsheet or table in an electronic lab notebook. The format matters less than whether the log is maintained and findable.

Changelogs

A changelog records what changed. It is most useful for documents, datasets, code outputs, protocols and resources that evolve over time.

A simple changelog should record:

  • version number or date;
  • what changed;
  • why it changed;
  • who made the change;
  • whether earlier outputs are affected.

For manuscripts or governance documents, the changelog may be part of a document control table. For code or data outputs, it may sit in a CHANGELOG.md file. For more technical projects, Git can provide a detailed history, but a short human-readable changelog is still useful for major changes15.

Environment notes

Environment notes document the software context required to rerun an analysis. They become particularly important when projects use R, Python, GIS software, command-line tools or specialist external software.

Environment notes should include:

  • programming language version, for example R 4.3.1 or Python 3.11;
  • key packages and versions;
  • operating system, if relevant;
  • specialist software or external dependencies;
  • hardware or processing assumptions, if relevant;
  • how to recreate the environment.

Example:

Software environment
- R version: 4.3.1
- Platform: Windows 11
- Key packages:
  - tidyverse 2.0.0
  - sf 1.0-14
  - lme4 1.1-35
  - targets 1.3.0

Environment management
- Project uses renv
- Lockfile: renv.lock stored in project root

To recreate:
install.packages("renv")
renv::restore()

This chapter only introduces environment notes. Chapter 7 covers code quality, checks and version-aware coding in more detail.

Electronic lab notebooks

Electronic Lab Notebooks (ELNs) provide a chronological record of work. They are common in laboratory settings but can also be useful for computational, qualitative and data-intensive research.

ELNs can help record:

  • what was attempted;
  • what happened;
  • unexpected problems;
  • decisions made during active work;
  • deviations from protocols;
  • links to files, outputs or scripts;
  • reflections on what to try next.

The Turing Way notes that ELNs can improve searchability, remote access, secure storage and sharing compared with traditional paper notebooks16. They are not, however, a replacement for all documentation. A project may still need a README, data dictionary, SOPs and formal governance records.

Avoid the “everything notebook” problem

ELNs are good for chronological work records. They are less good as the only place to store final instructions, data dictionaries, access rules or project orientation.

SOPs: documenting repeatable tasks

A Standard Operating Procedure is a set of step-by-step instructions for carrying out a routine task. SOPs are useful when a task needs to be done consistently across people, time or sites.

SOPs are most useful when:

  • the task is repeated;
  • inconsistency would affect data quality;
  • new people need training;
  • there are safety, governance or compliance implications;
  • the task involves multiple steps or decisions.

Hollmann and colleagues describe SOPs as tools for reducing deviations and maintaining consistent quality in research workflows17. For this workshop, the practical point is simple: write an SOP when a task needs to be done the same way more than once.

A good SOP should include:

  • title and purpose;
  • scope: when it applies and when it does not;
  • responsibilities;
  • required materials/software/access;
  • step-by-step procedure;
  • quality checks;
  • safety or governance considerations;
  • version, date and author;
  • review date.

Activity: the cup of tea protocol

Write a step-by-step protocol for making a cup of tea. You have three minutes.

Then compare protocols. Did everyone specify:

  • type and brand of tea?
  • cup, mug or teapot?
  • water source?
  • water temperature?
  • brewing time?
  • milk first or last?
  • how to judge when it is “done”?

This activity shows how expert knowledge can hide assumptions. The same problem occurs in research methods.

Expert bias and the curse of knowledge

Experts often omit details because those details feel obvious. This is one reason protocols can be difficult to reproduce. The person writing the protocol already knows the context, equipment, assumptions and tacit steps; the person reading it may not.

A useful test is:

Could a competent person who is new to the project follow this without asking you?

The answer does not need to be “yes” for every piece of documentation. But for SOPs, data dictionaries, README files and analysis instructions, it should be the aim.

Onboarding and offboarding

Onboarding and offboarding are documentation opportunities. New team members ask useful questions because they do not yet share the team’s assumptions. Leaving team members are the best people to document what only they know.

Onboarding documentation may include:

  • where research data are stored;
  • how to request access;
  • restrictions around data stewardship and sharing;
  • common software, packages and analysis tools;
  • key datasets and documentation;
  • authorship, publication and preprint expectations;
  • examples of previous ethics, governance or grant applications;
  • how to record decisions and issues.

Offboarding documentation may include:

  • where raw, derived and final data are stored;
  • who else has access;
  • ownership, custodianship and permitted reuse;
  • collaborators and contact points;
  • retention dates;
  • repository links, DOIs or publication links;
  • ethics, governance and data-sharing approvals;
  • software versions and hardware/software dependencies;
  • unfinished tasks and known issues.

The Turing Way provides useful onboarding and offboarding checklists that can be adapted for research groups and projects18.

Protocols, preregistration and reporting guidelines

Protocols and preregistrations are a special form of documentation. They make planned methods and analyses visible before results are known. They can help distinguish planned analyses from exploratory analyses, but they should not be treated as a prison. Deviations can be legitimate if they are documented transparently19,20.

Preregistration and protocol registration may be useful for:

  • clinical trials;
  • intervention studies;
  • systematic reviews;
  • planned secondary data analyses;
  • studies where analytic flexibility could affect interpretation.

Common platforms include OSF Registries, PROSPERO and protocols.io21–23. For systematic reviews, the Cochrane Handbook provides guidance on protocol development and review methods24. There are also templates specifically designed for secondary data analysis25.

Protocols can change

The value of a protocol is not that nothing changes. The value is that changes are visible, justified and interpretable.

Reporting guidelines are another form of transparency support. They help researchers report studies with enough detail for readers to understand what was done. EQUATOR curates reporting guidelines across study designs, including CONSORT, SPIRIT, STROBE, PRISMA, MOOSE and STARD26.

Reporting guidelines should usually be considered early, not only at manuscript submission.

Imagine you are contacted about a paper you published two years ago because a similar paper has reported contradictory results.

What could you provide to show your working?

Consider:

  • project README;
  • data dictionary or codebook;
  • metadata;
  • analysis scripts;
  • decision log;
  • changelog;
  • environment notes;
  • protocol or preregistration;
  • reporting guideline checklist;
  • data access statement;
  • repository or DOI;
  • documented deviations or sensitivity analyses.

Key takeaways

  • Documentation is how decisions remain visible.
  • Different documentation artefacts serve different purposes.
  • A README orientates; a codebook explains variables; metadata supports discovery and citation; logs record decisions and changes.
  • SOPs are useful for repeatable tasks; protocols and preregistrations make planned methods visible.
  • Proportionate documentation reduces risk, supports handover and strengthens reproducibility.
