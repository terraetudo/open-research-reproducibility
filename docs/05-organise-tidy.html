Chapter 5 — Organising and tidying data

Chapter 5

Organising and tidying data

This chapter moves from the project folder to the data file itself. The aim is to make datasets easier to check, analyse, document and reuse, without pretending that all real-world data arrive clean or simple.

What this chapter does

Chapter 3 focused on where project materials live: folders, filenames, raw and derived files, and lightweight versioning. Chapter 4 focused on the Data Management Plan as the record of decisions, responsibilities and constraints.

This chapter focuses on the structure and quality of the data files themselves. It covers common spreadsheet risks, tidy data principles, missing data, exclusions, data dictionaries and simple checks that make data easier to analyse and defend.

Learning objectives

By the end of this chapter, you should be able to:

  • explain why messy data make analysis fragile;
  • recognise common spreadsheet and data-entry risks;
  • describe the core principles of tidy data;
  • distinguish raw, cleaned and analysis-ready data without repeating the full project-organisation chapter;
  • document missingness, exclusions and filtering decisions;
  • identify minimum checks that should be carried out before analysis;
  • create or improve a simple data dictionary/codebook.
Boundary with other chapters

This chapter is not mainly about folder structures or file naming. Those are covered in Chapter 3. It is also not mainly about governance, storage or access. Those are covered in Chapter 4.

Here, the question is simpler: is the dataset itself structured so that it can be understood, checked and analysed safely?

If the data are messy, everything is fragile

Messy data make research harder to check, rerun, defend and reuse. Problems that look small during data entry can become substantial later, especially when data are linked, cleaned, merged or analysed by someone else.

Common consequences include:

  • analysing the wrong version of a dataset;
  • treating missing values as real values;
  • mixing text and numeric values in the same column;
  • losing information that was stored only in formatting or comments;
  • introducing manual corrections that cannot be reproduced;
  • making code more complicated than it needs to be.

Data Carpentry emphasises that small changes in spreadsheet formatting can improve efficiency and reliability when data are later cleaned and analysed1.

Quick reflection

Have you ever analysed, cleaned or shared the wrong version of a dataset?

If so, what made the error possible?

  • unclear filename;
  • mixed raw and cleaned data;
  • missing documentation;
  • manual edits;
  • several versions stored in different places;
  • no clear owner or approval point.

What makes a dataset usable?

A usable dataset is not necessarily perfect. It is one that is sufficiently clear, consistent and documented for its intended purpose.

A usable dataset should make the following visible:

Structure

Rows, columns and tables have clear meanings.

Meaning

Variables, units, categories and codes are documented.

Provenance

It is clear where the data came from and how they changed.

Quality

Missing values, exclusions and known limitations are recorded.

White and colleagues describe practical data-reuse principles that are also useful for the original research team: provide metadata, preserve an unprocessed form of the data, use standard formats, use good null values, and make the data easy to import and understand2.

Spreadsheet risks

Spreadsheets are useful for viewing, checking and small-scale data entry. They are also risky when they become the main place where cleaning, coding, analysis and documentation happen invisibly.

Common risks include:

  • mixed data types: numbers, dates and text in the same column;
  • hidden formatting: colours, font styles or cell comments used to convey meaning;
  • manual edits: changes made by hand without a script or log;
  • implicit missing values: blanks, NA, N/A, 0, -999 or text used inconsistently;
  • merged cells: useful for visual display but difficult for analysis software;
  • multiple tables in one sheet: visually convenient but machine-unfriendly;
  • multiple tabs with inconsistent structures: hard to combine and check later;
  • dates automatically changed by software.
Spreadsheet rule of thumb

Use spreadsheets for inspection and controlled data entry where appropriate. Avoid using formatting, colours, comments or manual edits as the only record of information that matters for analysis.

A minimum spreadsheet standard

Where spreadsheets are used, keep them boring and machine-readable.

A minimum standard is:

  • one table per sheet;
  • one header row;
  • one variable per column;
  • one observation per row;
  • no merged cells;
  • no blank rows or columns inside the data table;
  • no totals, graphs or notes embedded in the data table;
  • explicit and documented missing-value codes;
  • dates entered consistently, preferably using ISO format YYYY-MM-DD;
  • export or archive an analysis version as .csv where appropriate.

From human-readable to analysis-ready

A human-readable spreadsheet might combine address information in one cell:

address
170 Kessels Road Nathan Qld 4111 Australia
1 Parklands Dr, Southport QLD 4215

An analysis-ready table separates the components that may need to be checked, cleaned or linked:

street_number street_name street_type suburb state postcode country
170 Kessels Road Nathan Qld 4111 Australia
1 Parklands Drive Southport Qld 4215 Australia

This does not mean every dataset must be split into the smallest possible units. It means the structure should reflect how the data will be checked, analysed and reused.

Tidy data: the core idea

Tidy data is a simple but powerful way to structure rectangular datasets. Wickham summarises tidy data as a structure where each variable is a column, each observation is a row, and each type of observational unit forms a table3.

For practical purposes, start with these three rules:

One variable per column

Each column should contain one kind of information, such as date of birth, postcode, exposure value, or outcome status.

One observation per row

Each row should represent one unit of analysis: for example, one participant, one school, one measurement, one visit, or one time period.

One value per cell

Avoid storing multiple pieces of information in one cell when they may need to be analysed separately.

Tidy data are easier to search, filter, join, visualise and analyse. They also reduce the amount of special-purpose code needed to interpret the file.

Tidy does not mean final

A tidy dataset is not automatically correct, complete or analysis-ready. It still needs provenance, definitions, quality checks and documentation.

Raw, cleaned and analysis-ready data

Chapter 3 introduced the principle of separating raw, derived and output files. In this chapter, the same principle matters because a dataset can change through several states.

Raw data

The data as received, collected or extracted. These should be protected from overwrite.

Cleaned data

Data where formats, labels and obvious inconsistencies have been standardised. Cleaning steps should be scripted or documented.

Analysis-ready data

Data prepared for a specific analysis, with exclusions, derived variables and linkage decisions applied.

The important point is not the exact folder name. The important point is that someone can tell which version is closest to the original source, which version has been cleaned, and which version was used for a specific result.

Data provenance and lineage

Data provenance describes where data came from, what happened to them, and how outputs were produced. CESSDA frames data-file structure as part of organising and documenting research data so that files remain understandable over time4.

For each important dataset, you should be able to answer:

  • What is the original source?
  • Who provided or created it?
  • When was it received, extracted or generated?
  • What cleaning steps were applied?
  • What variables were derived?
  • What records were excluded?
  • Which script or workflow created the analysis-ready version?
  • Which outputs or publications used this version?

A simple data lineage record

flowchart LR
    A[Raw data received] --> B[Cleaning script]
    B --> C[Cleaned dataset]
    C --> D[Derivation / linkage]
    D --> E[Analysis-ready dataset]
    E --> F[Tables, figures, model outputs]

    B -. record decisions .-> G[Decision log]
    D -. record exclusions .-> G
    E -. describe variables .-> H[Data dictionary]

This is the minimum story someone needs in order to understand how the analysis dataset was produced.

Missing data and exclusions

Missing data are not just a technical nuisance. They can change the interpretation of results, the analytic sample and the conclusions.

Good practice includes:

  • using explicit and consistent missing-value codes;
  • distinguishing true zero from missing or not applicable;
  • documenting why data are missing when known;
  • recording exclusions and filtering decisions;
  • reporting how missingness was handled in analysis.

Kang describes missing data as a problem that requires attention to the pattern and mechanism of missingness, as well as the method used to handle it5. For this workshop, the main message is practical: do not let missingness become invisible.

Missing values need definitions

A blank cell, 0, NA, N/A, -999, unknown and not applicable do not mean the same thing unless you define them.

In R, NA is often appropriate for missing values. In a shared dataset or spreadsheet, the meaning of any missing-value code should be stated in the data dictionary.

Exclusions and filtering decisions

Exclusions should be visible and justifiable. This is particularly important when exclusions affect sample size, bias, precision or the interpretation of inequalities.

A minimum record should include:

Item Example
Exclusion rule Exclude records with missing postcode
Rationale Exposure assignment requires geocoding
Number excluded 184 records, 3.2% of sample
Timing Applied before exposure linkage
Script or decision record 02_clean_addresses.R, decision log entry 2025-02-12
Sensitivity check Results compared with missing-postcode group described separately
Avoid hidden exclusions

Do not delete rows manually and leave no trace. Use a scripted filter where possible, and record the rule and rationale in a decision log, README or analysis note.

Data dictionaries and codebooks

A data dictionary or codebook explains what each variable means. It is one of the most useful low-burden reproducibility tools.

A simple data dictionary should include:

  • variable name;
  • short description;
  • source dataset;
  • data type;
  • units, if relevant;
  • allowed values or categories;
  • missing-value codes;
  • derivation rule, if the variable was created;
  • notes or known limitations.

Minimal data dictionary example

variable description source type allowed values / units missing code notes
response_id Unique survey response identifier Census 2021 character 17-character string blank Not personally identifying
postcode Residential postcode at survey date Survey character Valid UK postcode NA Standardised before linkage
pm25_annual Annual mean PM2.5 exposure Modelled exposure numeric µg/m³ NA Linked by postcode centroid
excluded Record excluded from main analysis Derived logical TRUE / FALSE none Rationale in exclusion_reason

Basic checks before analysis

Some checks are simple but high-value. They do not require a formal testing framework.

Before analysis, ask:

  • Does the dataset have the expected number of rows and columns?
  • Are required variables present?
  • Are key variables the expected type?
  • Are dates in one format?
  • Are numeric variables within plausible ranges?
  • Are categorical variables using expected labels?
  • Are missing values coded consistently?
  • Are duplicate records expected or accidental?
  • Has the analytic sample size changed since the previous version?

These checks belong naturally in the code-quality chapter as well. Here, the focus is on what to check in the data. Chapter 7 will cover how to implement these checks in R code.

Messy data activity

Activity: identify the risks

Look at the example below and identify what would make it difficult to analyse safely.

Date Age Where was the article Blood type Preferred Gender
23/1/2023 32 United States AB Female
Feb-17 54 A N/A
2/2/2000 12 USA 0 Male
23-Mar-93 53 Brisbane, Australia #NAME? Mail

Questions:

  • Are dates recorded consistently?
  • Is Age the right variable, or would date of birth plus date of measurement be more stable?
  • Are locations recorded consistently?
  • Is 0 a valid blood type, a missing code, or an error?
  • What does N/A mean here?
  • Is Mail a spelling mistake or a valid category?
  • What has Excel or spreadsheet software changed automatically?
Possible fixes
  • Use a consistent date format, preferably YYYY-MM-DD.
  • Consider recording date of birth rather than age if age depends on timing.
  • Split location into structured fields where useful, such as country, region and city.
  • Define missing-value codes explicitly.
  • Use controlled categories for gender and blood type where appropriate.
  • Keep a data dictionary so future users understand all valid codes.

Minimum viable practice

For most projects, a realistic minimum is:

Protect the raw data

Keep a copy of the data as received. Do not overwrite it.

Use one analysis dataset

Make clear which cleaned or analysis-ready file was used for each analysis.

Create a data dictionary

Document variables, units, categories and missing codes.

Record exclusions

State what was removed, why, when and by whom.

Beginner and advanced next steps

Choose one action

Beginner

  • Decide how missing values will be represented.
  • Use YYYY-MM-DD for dates.
  • Create a short data dictionary for one dataset.
  • Save an analysis-ready copy separately from the raw data.

Advanced

  • Add automated checks for row counts, variable names and plausible ranges.
  • Create a scripted cleaning pipeline from raw to analysis-ready data.
  • Add exclusion counts to the analysis report.
  • Use controlled vocabularies or metadata standards where appropriate.

Key takeaways

  • Messy data make analysis fragile, even when the project folder is well organised.
  • Spreadsheets are useful but risky when formatting, comments or manual edits carry analytical meaning.
  • Tidy data principles make datasets easier to check, analyse and reuse.
  • Missing values and exclusions need explicit definitions and records.
  • A data dictionary is one of the simplest and most valuable reproducibility tools.
  • Chapter 7 will show how to implement some of these checks in R code.
