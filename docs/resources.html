Resources and Further Reading

This chapter gathers together tools, UCL support, further reading, and copy-ready templates referenced across Chapters 1–9.
It is intended as a one-stop practical library for early-career researchers and researchers onboarding to projects at UCL.

Use this chapter when you need:

  • examples to implement ideas from the Chapters;
  • templates to copy into your own repositories;
  • links to UCL policies, services, and support.
How to use this page
  • Start here if you are new to UCL or new to a project team.
  • Use the checklists first; use the deeper links when you need more detail.
  • Sections marked Advanced / reference are optional unless your project context requires them.

Onboarding: first actions on a new project

Use this when you join a project (or onboard someone else).

  1. Find or create the project README (purpose, scope, structure, contacts).
  2. Confirm data access and governance (what you can/can’t do; where it’s documented).
  3. Identify the source of truth (repository/shared drive) and the agreed naming/versioning rules.
  4. Locate raw data and confirm it is immutable; find how derived data are produced.
  5. Identify the analysis entry point (main script / workflow) and the environment notes.
  6. Find the decision log / ELN and record your first assumptions and uncertainties.
  7. Check expected outputs (papers, datasets, code releases) and likely release points.

Derived and intermediate data: retention decision prompts (quick guide)

Use these prompts when writing the DMP sections on retention, preservation, and storage.

Record explicitly:

  • which derived datasets are considered analysis-critical (needed to reproduce results);
  • which intermediate files are recomputable (can be regenerated from scripts + inputs);
  • what will be retained during the project vs after project end;
  • expected access model (open / safeguarded / controlled / mediated) for each retained component;
  • how provenance will be captured at a high level (e.g., “derived datasets are generated by scripted workflows and versioned”).

Chapter 5 covers how to implement raw/derived organisation and provenance in practice; the DMP should state the commitment, scope, and rationale.

Essential metadata (minimum standard)

Supports Chapter 5 — Organising and tidying data
Referenced by Chapters 6 (documentation) and 10 (publishing and reuse).

Essential Metadata Checklist (minimum standard)

When documenting a dataset, code release, protocol, or other output, include at least:

  • Title
  • Creators / contributors (including ORCID where possible)
  • Dates (created; last updated)
  • Description (what it is; why it exists; what it can and cannot support)
  • Version
  • Methods / provenance summary (how it was generated; key transformations)
  • Geography / time coverage (if relevant)
  • File formats and any required software
  • Access conditions (open / controlled / mediated; who to contact)
  • Licence (or explicit statement if licensing is constrained)
  • Funding / grant (if relevant)
  • Persistent identifier (DOI where appropriate; or stable URL + repository record)
Minimum rule of thumb

If a competent colleague can answer what it is, where it came from, how to access it, how to cite it, and what they are allowed to do with it, your metadata is likely “good enough” for most purposes.

File naming and versioning (minimum standard)

Recommended elements

  • short descriptive name
  • date (ISO): YYYYMMDD
  • version identifier (e.g. v01)
  • no spaces or special characters

Examples

survey_responses_raw_20240315_v01.csv
survey_responses_clean_20240318_v02.csv
analysis_model1_20240402_v03.R
protocol_amendment_20240510_v01.docx

Versioning cheat-sheet

Create a new version when:

  • cleaning/transformation changes;
  • analysis logic/parameters change;
  • outputs are regenerated using modified inputs;
  • errors are corrected.

Record:

  • version identifier;
  • date;
  • what changed (1–2 lines);
  • who changed it.

Optional: code version control (Git) (Advanced / reference)

Appropriate when code is central and/or multiple contributors edit scripts.

Key principles:

  • commit little and often with meaningful messages;
  • avoid committing large data files;
  • use tags/releases for milestones.

Reporting guidelines for health and population research

This section provides orientation and signposting. Detailed application and worked examples belong in Chapter 10.

High-quality research is only useful if it is reported transparently. Reporting guidelines specify the minimum information needed to describe how a study was designed, conducted, analysed, and interpreted.

More than 250 reporting guidelines exist, curated by the EQUATOR Network (Enhancing the QUAlity and Transparency Of health Research).

EQUATOR Network — reporting guidelines library10

Core guideline families (commonly used):

  • CONSORT — randomised controlled trials
  • SPIRIT — protocols for interventional trials
  • STROBE — observational studies (cohort, case–control, cross-sectional)
  • PRISMA — systematic reviews and meta-analyses (incl. extensions such as PRISMA-S)
  • MOOSE — meta-analyses of observational studies
  • STARD — diagnostic accuracy studies

Practical licensing rule

Pick a licence per output type (text, data/metadata, code) and record it alongside access conditions. If you cannot licence something, record why and what can be shared instead (e.g., metadata + code + synthetic example).

Repository “minimum viable release” guidance

When you release code/data/materials to a repository, include at least:

  • README (purpose, how to run/use, inputs/outputs)
  • CITATION information (e.g., CITATION.cff)
  • Licence file
  • Version tag/release notes (what changed)
  • Environment notes (how to recreate the software environment)
  • Data documentation and access statement (even if data are restricted)
